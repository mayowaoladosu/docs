# Nubis Launch Phase 1

`Nubis Launch` is the first control-plane slice for application hosting on top of Nubis infrastructure. Phase 1 focuses on the deploy loop for `web`, `worker`, and `cron` services with explicit lifecycle states, operator-visible events, and a GitHub-first source configuration flow.

## Scope

- Private-beta access is enforced per organization through `launch_org_access`
- Launch projects reuse the existing `projects` table and attach Launch-specific records through dedicated `launch_*` tables
- Services can be created with `github` metadata or a raw `repo_url`
- Deployments move through an explicit state machine:
  - `queued`
  - `building`
  - `build_failed`
  - `image_ready`
  - `provisioning`
  - `starting`
  - `unhealthy`
  - `live`
  - `rollback_in_progress`
  - `rolled_back`
  - `canceled`
- Build and runtime execution are handled by concrete Phase 1 adapters that record OCI image output and Kubernetes rollout metadata for the control-plane workflow

## Backend Layout

- `services/api-gateway/src/launch/models.rs`
  Launch domain models and string-backed enums
- `services/api-gateway/src/launch/state_machine.rs`
  Explicit deployment lifecycle validation
- `services/api-gateway/src/launch/repository.rs`
  Persistence for Launch access, services, deployments, events, logs, and build metadata
- `services/api-gateway/src/launch/service.rs`
  Input validation, private-beta gating, env var encryption, and Launch orchestration entry points
- `services/api-gateway/src/launch/adapters.rs`
  Build adapter and runtime adapter abstractions plus the Phase 1 Dockerfile/Kubernetes implementations
- `services/api-gateway/src/workers/launch_deployments.rs`
  Background deployment worker that advances queued deployments through the state machine
- `services/api-gateway/src/handlers/launch.rs`
  Public Launch API surface

## Frontend Layout

- `new-frontend/src/lib/launch.ts`
  Launch API client and TypeScript models
- `new-frontend/src/components/launch/*`
  Launch status, timeline, access gate, and log UI building blocks
- `new-frontend/src/pages/launch/*`
  Project list, project detail, service detail, and deployment detail pages

## API Surface

All Launch routes are scoped under an organization:

- `GET /api/v1/orgs/:org_id/launch/access`
- `GET /api/v1/orgs/:org_id/launch/projects`
- `POST /api/v1/orgs/:org_id/launch/projects`
- `GET /api/v1/orgs/:org_id/launch/projects/:project_id`
- `GET /api/v1/orgs/:org_id/launch/projects/:project_id/services`
- `POST /api/v1/orgs/:org_id/launch/projects/:project_id/services`
- `GET /api/v1/orgs/:org_id/launch/services/:service_id`
- `GET /api/v1/orgs/:org_id/launch/services/:service_id/deployments`
- `POST /api/v1/orgs/:org_id/launch/services/:service_id/deployments`
- `GET /api/v1/orgs/:org_id/launch/services/:service_id/logs`
- `GET /api/v1/orgs/:org_id/launch/deployments/:deployment_id`
- `GET /api/v1/orgs/:org_id/launch/deployments/:deployment_id/events`
- `GET /api/v1/orgs/:org_id/launch/deployments/:deployment_id/logs`

## Local Setup

1. Start the API gateway with the usual Nubis environment.
2. Run migrations so `425_nubis_launch_phase1.sql` is applied.
3. Grant Launch access to a development org:

```sql
\i scripts/seed_launch_beta_access.sql
```

Update the placeholder org UUID in that script before running it.

4. Start the frontend and open `/launch`.

## Notes

- Environment variable values are encrypted before storage using a key derived from the configured server secret plus the organization ID.
- Phase 1 models GitHub as the primary source mode through repository metadata fields. Full OAuth installation and webhook automation are outside this phase boundary.
- The runtime target remains Kubernetes, and build output is normalized to OCI image references recorded on the deployment and build records.
