# Nubis Launch Phase 1-2

`Nubis Launch` is the first control-plane slice for application hosting on top of Nubis infrastructure. Phase 1 focuses on the deploy loop for `web`, `worker`, and `cron` services with explicit lifecycle states, operator-visible events, and control-plane persistence. Phase 2 adds the GitHub-first connection flow, synced repository selection, repo-scoped webhook installation, push-triggered deployment creation, preview environments, custom domains, rollback execution, OCI registry publishing, and Kubernetes runtime apply/health integration.

## Scope

- Private-beta access is enforced per organization through `launch_org_access`
- Launch projects reuse the existing `projects` table and attach Launch-specific records through dedicated `launch_*` tables
- Services can be created with `github` metadata or a raw `repo_url`
- Deployments move through an explicit state machine:
  - `queued`
  - `building`
  - `build_failed`
  - `image_ready`
  - `provisioning`
  - `starting`
  - `unhealthy`
  - `live`
  - `rollback_in_progress`
  - `rolled_back`
  - `canceled`
- Build and runtime execution are handled by concrete Phase 1 adapters that record OCI image output and Kubernetes rollout metadata for the control-plane workflow

## Backend Layout

- `services/api-gateway/src/launch/models.rs`
  Launch domain models and string-backed enums
- `services/api-gateway/src/launch/state_machine.rs`
  Explicit deployment lifecycle validation
- `services/api-gateway/src/launch/repository.rs`
  Persistence for Launch access, services, deployments, events, logs, and build metadata
- `services/api-gateway/src/launch/service.rs`
  Input validation, private-beta gating, env var encryption, and Launch orchestration entry points
- `services/api-gateway/src/launch/adapters.rs`
  Build adapter and runtime adapter abstractions plus the Phase 1 Dockerfile/Kubernetes implementations
- `services/api-gateway/src/launch/github.rs`
  GitHub OAuth completion, encrypted connection storage, repository sync, webhook installation, and push-triggered deployment orchestration
- `services/api-gateway/src/workers/launch_deployments.rs`
  Background deployment worker that advances queued deployments through the state machine
- `services/api-gateway/src/handlers/launch.rs`
  Public Launch API surface
- `services/api-gateway/src/handlers/launch_github.rs`
  GitHub connection endpoints, OAuth callback handling, and Launch webhook ingress

## Local Setup

1. Start the API gateway with the usual Nubis environment.
2. Set the Launch GitHub environment variables:

```bash
GITHUB_CLIENT_ID=...
GITHUB_CLIENT_SECRET=...
API_URL=http://localhost:8080
FRONTEND_URL=http://localhost:5173
LAUNCH_REGISTRY_HOST=registry.usenubis.com
LAUNCH_REGISTRY_NAMESPACE=launch
LAUNCH_REGISTRY_USERNAME=launch-builder
LAUNCH_REGISTRY_PASSWORD=...
LAUNCH_BUILDER_WORKDIR=/tmp/nubis-launch-builds
```

3. Run migrations so both Launch migrations are applied:

```text
425_nubis_launch_phase1.sql
426_nubis_launch_phase2_github.sql
427_nubis_launch_phase2_2_runtime.sql
```

4. Grant Launch access to a development org:

```sql
\i scripts/seed_launch_beta_access.sql
```

Update the placeholder org UUID in that script before running it.

5. In the GitHub OAuth app settings, set the callback URL to:

```text
http://localhost:8080/api/v1/launch/github/callback
```

6. Start the frontend and open `/launch`.

## Notes

- Environment variable values are encrypted before storage using a key derived from the configured server secret plus the organization ID.
- GitHub is now the primary Launch source path. Users connect GitHub first, sync repositories into the organization cache, and Launch installs a repository-scoped push webhook when a GitHub-backed service is created.
- Repository URL entry remains available as the manual fallback path for advanced or disconnected workflows.
- Push webhooks create Launch deployments only for services whose tracked branch matches the pushed ref, and delivery handling is idempotent per service.
- Dockerfile builds run through the Docker CLI, log into the configured registry, push OCI images, and record the pushed digest back onto the Launch deployment.
- The runtime target remains Kubernetes. Web services are applied as `Deployment + Service + Ingress`, workers as `Deployment`, and cron services as `CronJob`.
- Preview branches create isolated Launch preview environments and receive their own runtime URLs.
- Custom domains are written through Cloudflare DNS and reconciled against the live Kubernetes ingress/TLS state so the dashboard reflects pending DNS, pending TLS, and live status from the runtime.
- Rollbacks reuse the previously stable OCI image reference instead of rebuilding the older revision.
