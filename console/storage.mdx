## Storage on Nubis

Nubis gives you two primary storage models:

- **Volumes** for low-latency block storage attached to Instances.
- **Spaces** for S3-compatible object storage used by assets, backups, logs, media, and static content.

Use this page as the decision guide before you create either one.

## Choose the Right Product

### Use Volumes when you need

- Extra disk capacity for an Instance without rebuilding it.
- Persistent data for databases, queues, search indexes, or file processing jobs.
- Predictable block-device semantics such as partitioning, formatting, and mounting.
- Expansion of storage independently from compute size.

### Use Spaces when you need

- Durable object storage for uploads, archives, logs, media, backups, or static files.
- Public or private file delivery over HTTPS.
- CDN-backed asset distribution.
- S3-compatible tooling and SDK workflows.

## Architecture Guidance

### Volumes

Volumes are best when your workload expects a filesystem mounted inside Linux. They should be treated as application data disks, not as a replacement for object storage or multi-writer shared filesystems.

Recommended pattern:

1. Keep the root disk for OS and bootstrap packages.
2. Put mutable application data on a dedicated Volume.
3. Use snapshots or application-level backups for recovery.
4. Resize storage independently as the dataset grows.

### Spaces

Spaces are best when your workload stores blobs rather than mounted filesystems. Application uploads, rendered media, Terraform state, static assets, export archives, and log sinks all fit naturally here.

Recommended pattern:

1. Store unstructured files in Spaces.
2. Serve public assets through the CDN when latency matters.
3. Use versioning for rollback-sensitive buckets.
4. Keep private data behind scoped access keys and bucket policies.

## Operations Checklist

Before production launch, confirm the following:

- Region choice matches where your Instances, databases, and users are located.
- Backup and recovery expectations are defined before data lands in production.
- Access control is scoped to teams, service accounts, or application-specific credentials.
- Cost ownership is clear by project and by environment.
- Lifecycle rules or retention policies are configured for data that grows continuously.

## Cost and Billing

Storage cost is easiest to manage when each product is treated differently:

- **Volumes** scale primarily with provisioned capacity.
- **Spaces** scale with stored data, requests, and egress-sensitive delivery patterns.
- **CDN traffic, snapshots, backups, and cross-service replication** should be reviewed separately from base storage cost.

## Operational Best Practices

- Keep production data in dedicated projects with clear ownership.
- Tag storage resources by environment, team, and workload.
- Do not mix hot database storage and archival object storage in the same operating model.
- Test restore procedures, not just create procedures.
- Review growth trends monthly so storage expansion does not happen as an emergency task.
